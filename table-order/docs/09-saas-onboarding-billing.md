# 09. SaaS 가입·온보딩·플랜·슈퍼어드민

- 버전: v0.1 (2026-07-12)
- 구현 소유: `auth-tenancy` 에이전트 (`app/(platform)`, `src/auth`, `middleware.ts`). 구독 결제는 `payments`(docs/08 §5), 플랜 게이트 상수는 `packages/shared`.

## 1. 가입 퍼널

```
랜딩(/) → 데모 룩북 체험(/s/demo) → 가입(/join) → 온보딩 위저드(/onboarding) → 개시 체크리스트 → 첫 주문
```

- 랜딩: 히어로에 실제 룩북 모션 데모, 가격, "하드웨어 없이 오늘 개통" 소구. 데모 매장(`demo` 시드)은 읽기 전용 + 가짜 주문 체험 모드(POS 데모 화면 분할 시연).
- 가입: 이메일+비밀번호(Supabase Auth) → `POST /api/platform/signup` 트랜잭션: authUser + StaffUser(OWNER) + Store(ONBOARDING) + Subscription(TRIAL, trialEndsAt=+14d). 슬러그는 매장명 기반 자동 제안 + 중복 검사(예약어: demo, admin, api, super 등 차단).

## 2. 온보딩 위저드 (5단계, 각 단계 저장·중단 후 재개 가능)

| 단계 | 내용 | 완료 조건 |
|---|---|---|
| ① 매장 정보 | 이름·업종·전화·주소·영업시간 | 필수값 |
| ② 테이블 | 테이블 수 입력 → 일괄 생성(라벨 자동 1..N, 수정 가능) → **QR 시트 PDF 다운로드** | 테이블 ≥ 1 |
| ③ 메뉴 | 카테고리 템플릿 제안(업종별) → 메뉴 3개 이상 등록(사진 업로드 포함, "나중에" 허용하되 개시 체크리스트에 남김) | 메뉴 ≥ 1 |
| ④ 룩북 테마 | 커버 업로드, 팔레트/서체 프리셋 선택 → 실시간 미리보기 | 기본값 허용 |
| ⑤ 개시 | 미리보기 최종 확인 → `개시하기` → Store.status ACTIVE | — |

- 개시 체크리스트(대시보드 상단 카드로 잔존): 메뉴 5개↑, 전 메뉴 사진, QR 인쇄 확인, 테스트 주문 1회(자기 폰 스캔 가이드), 직원 초대.
- 온보딩 중에도 `/s/[slug]`는 ONBOARDING 상태면 "준비 중" 커버만 노출.

## 3. 플랜 & 기능 게이트

가격은 **가설**(출시 전 시장 조사로 확정, 여기선 게이트 구조 확정이 목적):

| | TRIAL (14일) | BASIC ₩29,000/월 | PRO ₩59,000/월 |
|---|---|---|---|
| 테이블 | 30 | 15 | 무제한 |
| 메뉴 | 무제한 | 50 | 무제한 |
| 사진/메뉴 | 5 | 3 | 10 |
| 선결제(PG) | ✅ | ❌ | ✅ |
| 통계 | 기본 | 기본 | 고급(시간대·회전율) |
| 테마 | 전체 | 프리셋만 | 전체 + 포컬포인트 |
| 직원 계정 | 5 | 3 | 무제한 |
| AI 연출컷 크레딧 | 10회(체험) | 애드온 구매 | 50회/월 |

- 구현: `packages/shared/src/contracts/platform.ts`의 `PLAN_LIMITS: Record<PlanType, PlanLimits>` 단일 상수. 서버는 생성 API에서 한도 검사(`PLAN_LIMIT_EXCEEDED`), POS는 게이지·업그레이드 CTA 표시. 게이트 검사 헬퍼 `assertWithinPlan(ctx, "tables" | "items" | "aiCredits" | ...)`는 backend-api가 제공(AI 크레딧의 차감·실패 환불 정합은 ai-imagery 소유, docs/12 §6).
- 다운그레이드 시 초과분은 삭제하지 않고 **읽기 전용 잠금**(신규 생성만 차단) — 데이터 파괴 금지 원칙.
- 구독 상태별 접근 매트릭스: TRIALING/ACTIVE=전체, PAST_DUE=전체+배너, SUSPENDED=POS 결제설정만 접근 가능·룩북은 안내 페이지(docs/08 §5 dunning).

## 4. 인증·역할 체계 (auth-tenancy 구현 기준)

| 주체 | 인증 | 권한 |
|---|---|---|
| 손님 | 무인증 (`mb_table` 서명 쿠키, 3h) | 해당 매장 룩북·자기 세션 주문/호출 |
| STAFF | Supabase Auth 세션 | 주문 보드·품절 토글·정산·호출 처리 |
| MANAGER | 〃 | + 메뉴/테이블/QR 관리, 강제 종료, 환불 |
| OWNER | 〃 | + 매장 설정·테마·직원·구독·통계 |
| PLATFORM_ADMIN | 별도 allowlist(PlatformAdmin 테이블) | `/super` 전용 |

- 한 계정이 여러 매장 소속 가능(StaffUser 다중 행) → 로그인 후 매장 선택기. `requireStaff(slug, minRole)`가 멤버십·역할 검사 후 `tenantCtx` 반환(docs/02 §5).
- 초대 플로우: OWNER가 이메일 입력 → 초대 토큰 메일 → 수락 시 계정 생성/연결 + StaffUser 생성.

## 5. 슈퍼어드민 (`/super`)

- 매장 테이블: 검색/필터(status, plan) — 이름, slug, 플랜, 상태, 최근 7일 주문수, 가입일.
- 매장 상세: 지표 스냅샷, 구독 이력, AuditLog 열람, 액션(SUSPENDED 전환·해제, 트라이얼 연장, 임시 로그인 링크는 **비도입** — 지원은 화면공유로).
- 플랫폼 지표 대시보드: 가입 퍼널(가입→개시→첫주문), MRR, 이탈. MVP는 숫자 카드 수준.
