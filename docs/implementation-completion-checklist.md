# 사내 근태 MVP 기능 완성 체크리스트

작성일: 2026-07-08  
범위: 직원 셀프서비스, 관리자 정책 운영, 파일럿 검증

## 1. 기능별 완료 기준

### 1.1 직원 셀프서비스

- 직원은 모바일 첫 화면에서 오늘 출근 상태, 다음 행동, 휴가/야근/정정 대기 상태를 확인할 수 있다.
- 직원은 GPS, QR, GPS 실패 허용 흐름으로 출근/퇴근을 완료할 수 있고 결과 상태를 즉시 확인할 수 있다.
- 직원은 본인의 근태 기록, 조기퇴근 누계, 휴가 잔액, 야근 신청 내역, 급여명세서만 조회할 수 있다.
- 직원이 휴가, 야근, 근태 정정 요청을 제출하면 상태가 `PENDING`으로 저장되고 감사 로그가 남는다.
- 본인 외 직원의 근태, 휴가, 야근, 급여명세서 데이터는 직원 화면과 API 응답에서 노출되지 않는다.

### 1.2 근태/보정

- 출근/퇴근 기록은 일자와 직원 기준으로 생성 또는 갱신되며 GPS 판정, QR 보조 인증, GPS 실패 허용 상태를 구분해 저장한다.
- 17:00 이전 퇴근은 조기퇴근 분으로 계산되어 조기퇴근 장부에 반영된다.
- GPS 실패, 위치 권한 거부, 누락 출퇴근 등 예외는 정정 요청 또는 관리자 보정으로 처리할 수 있다.
- 관리자 보정은 원본 기록을 삭제하지 않고 전/후 값, 사유, 보정자, 생성 시각을 별도 이력으로 남긴다.
- 관리자 대시보드에는 신규 보정 이력이 즉시 표시되고 감사 로그에서 추적 가능해야 한다.

### 1.3 휴가 승인

- 직원이 연차, 반차, 특별휴가, 무급휴가를 신청하면 승인 대기 목록에 표시된다.
- 승인자가 휴가를 승인 또는 반려하면 해당 신청 상태가 갱신되고 감사 로그가 남는다.
- 승인 완료된 휴가 신청은 `pendingLeaveRequests`에서 제외된다.
- 휴가 잔액은 법정 연차와 회사 선사용 휴가를 구분해 계산하고, 직원에게는 사용 가능 합계를 먼저 보여준다.
- 승인 대기 휴가는 잔여 휴가 표시에서 별도 예정 차감분으로 확인할 수 있다.

### 1.4 야근/상계/수당 인정

- 직원이 야근을 신청하면 기본값은 승인 대기이며 수당 인정값은 `false`로 저장된다.
- 승인자가 야근을 승인하면 승인 상태가 저장되고 직원 스냅샷의 활성 야근 상계 계산 대상이 된다.
- 조기퇴근 분이 있으면 승인 야근 시간에서 우선 상계하고, 잔여 야근 분만 수당 인정 후보가 된다.
- 관리자가 `payApproved=true`로 수당 집계 인정을 처리하면 `employeeSnapshot.overtimeOffset.payEligibleMinutes`가 갱신된다.
- 성수기 등 상계 제외 정책은 별도 상태값으로 표시하고 자동 급여 계산으로 바로 연결하지 않는다.

### 1.5 급여명세서

- HR 관리자는 직원별 월 급여명세서 메타데이터를 업로드할 수 있고 업로드 감사 로그가 남는다.
- 직원은 본인의 삭제되지 않은 급여명세서만 목록에서 볼 수 있다.
- 급여명세서 삭제는 물리 삭제가 아니라 `deletedAt`을 기록하는 soft delete로 처리한다.
- soft delete된 명세서는 직원 스냅샷의 `payrollStatements`와 관리자 활성 목록에서 제외된다.
- 급여명세서 조회, 다운로드, 업로드, 삭제는 민감 이벤트로 감사 로그에 남긴다.

### 1.6 설정/감사 로그

- 근무지 좌표, 허용 반경, QR 사용 여부, 기본 근무시간, 성수기 정책은 설정값으로 분리한다.
- 권한은 직원, 승인자, HR 관리자, 시스템 관리자로 구분하고 역할별 API 접근 범위를 제한한다.
- 출퇴근 실패 허용, 관리자 보정, 휴가/야근 결정, 수당 인정, 급여명세서 변경은 감사 로그로 추적한다.
- 감사 로그에는 행위자, 대상 유형, 대상 ID, 시각, 액션, 사유 또는 상세 설명이 포함된다.
- 일반 운영 API로 감사 로그를 수정하거나 삭제할 수 없어야 한다.

### 1.7 파일럿 운영

- 파일럿 대상 직원과 전체 직원 수를 대시보드에서 구분해 볼 수 있다.
- 운영팀 파일럿 기간 동안 GPS 실패 빈도, QR 대체 인증, 보정 건수, 휴가/야근 승인 처리 시간을 주간 단위로 점검한다.
- 파일럿 종료 전 직원 셀프서비스, 관리자 승인, 급여명세서 권한, 감사 로그 시나리오를 전부 재검증한다.
- 정책 예외와 대표 의사결정 필요 항목은 파일럿 회고 문서에 남기고 운영 전 기본값을 확정한다.
- 기능 출시 기준은 `npm run test` 통과, 정책 회귀 테스트 통과, 주요 화면 수동 QA 완료로 한다.

## 2. 대표 의사결정 필요 항목과 CTO 기본 구현값

| 기능 | 대표 의사결정 필요 항목 | CTO 기본 구현값 |
|---|---|---|
| 직원 셀프서비스 | 직원에게 GPS 실패 허용 버튼을 즉시 노출할지, QR 인증을 먼저 유도할지 결정 | GPS 실패 시 QR 인증을 우선 노출하고, 재시도/정정 요청을 같은 화면에 제공 |
| 근태/보정 | 인정지각/인정조퇴 승인 권한을 팀장까지 줄지 HR 관리자에게만 둘지 결정 | HR 관리자 이상이 관리자 보정을 수행하고, 승인자는 정정 요청 결정만 처리 |
| 휴가 승인 | 선사용 휴가 월 1일 정책의 상한과 입사 초기 적용 조건 확정 | 입사 3개월 후 월 1일, 파일럿 기간에는 잔액 표시와 신청 검증까지만 구현 |
| 야근/상계/수당 인정 | 조기퇴근 상계 제외 기간과 수당 인정 최종 승인자를 결정 | 1월, 2월은 상계 제외 성수기로 두고, 수당 인정은 HR 관리자 액션으로 분리 |
| 급여명세서 | 급여명세서 보관 기간, 삭제 사유 필수 여부, 다운로드 로그 범위 확정 | 삭제는 soft delete로 시작하고, 조회/다운로드/삭제는 모두 감사 로그 대상 |
| 설정/감사 로그 | 원시 위치 좌표 저장 여부와 보관 기간 확정 | 원시 좌표는 저장하지 않고 거리, 정확도, 근무지 매칭 결과만 저장 |
| 파일럿 운영 | 파일럿 성공 기준과 전사 전환 시점을 확정 | 운영팀 파일럿으로 시작하고 GPS 실패율, 보정률, 승인 처리 누락을 주간 점검 |

## 3. 정책 회귀 테스트 기준

- 휴가 승인 후 승인된 신청이 `pendingLeaveRequests`에서 제외되는지 확인한다.
- 야근 승인과 수당 인정 후 직원 스냅샷의 `overtimeOffset.payEligibleMinutes`가 갱신되는지 확인한다.
- 근태 보정 생성 후 관리자 대시보드의 `corrections`에 신규 보정이 표시되는지 확인한다.
- 급여명세서 soft delete 후 직원 스냅샷의 `payrollStatements`에서 삭제된 명세서가 제외되는지 확인한다.
