# 사내 근태·휴가·급여 확인 웹서비스 PRD

작성일: 2026-07-08  
대상: 내부 직원용 모바일 우선 웹서비스  
목표: 직원이 휴대폰으로 출퇴근, 휴가/연차/반차 신청, 야근 신청, 급여명세서 확인을 간단히 처리하고, 관리자는 승인·정산·현황 확인을 빠르게 수행한다.

## 1. 제품 방향

### 1.1 핵심 원칙

- 모바일 우선: 직원의 주요 행동은 휴대폰에서 10초 안에 완료되어야 한다.
- GPS 우선 인증: 출퇴근은 GPS 허용 반경 검증을 기본으로 한다.
- QR/Wi-Fi 보조 인증: GPS 오류, 실내 오차, 권한 문제를 보완한다.
- 법정 연차와 회사 선사용 휴가를 분리한다.
- 오버엔지니어링 배제: 초기는 단순한 승인 흐름과 명확한 장부 중심으로 구현한다.
- 추후 확장 가능성: 비즈니스 로직은 도메인 모듈 단위로 분리한다.

### 1.2 MVP 범위

- 직원 로그인 및 권한
- 모바일 출근/퇴근 처리
- GPS 메인 인증, QR/Wi-Fi 보조 인증
- GPS 수신 실패 시 마킹 후 QR 또는 버튼 클릭으로 출퇴근 허용
- 출퇴근 누락·오류 정정 요청
- 관리자 출퇴근 시간 보정, 인정지각, 인정조퇴
- 연차, 반차, 휴가 신청 및 승인
- 입사 첫해 선사용 휴가 정책
- 야근 신청 및 승인
- 조기퇴근 시간 월별/기간별 조회
- 급여명세서 조회
- 관리자 대시보드
- 감사 로그

### 1.3 비범위

- 급여 자동 계산 전체 시스템
- 회계/ERP 직접 연동
- 복잡한 교대근무 스케줄러
- 실시간 위치 추적
- 얼굴인식/생체인증
- 과도한 조직도/전자결재 워크플로우

## 2. 사용자와 권한

### 2.1 역할

| 역할 | 설명 | 주요 권한 |
|---|---|---|
| 직원 | 일반 사용자 | 본인 출퇴근, 신청, 조회 |
| 팀장/승인자 | 신청 승인 담당 | 소속 직원 신청 승인/반려 |
| HR 관리자 | 인사 운영 담당 | 직원, 정책, 근태, 휴가, 급여명세서 관리 |
| 시스템 관리자 | 서비스 운영 담당 | 권한, 설정, 감사 로그, 근무지 관리 |

### 2.2 권한 원칙

- 직원은 본인 데이터만 조회한다.
- 팀장/승인자는 승인 대상 직원의 신청과 요약 데이터만 조회한다.
- HR 관리자는 전체 근태/휴가/급여명세서를 관리한다.
- 급여명세서는 민감정보로 취급하여 접근 로그를 반드시 남긴다.

## 3. 핵심 사용자 흐름

### 3.1 직원 출근

1. 직원이 모바일 웹에 접속한다.
2. 홈 화면에서 `출근` 버튼을 누른다.
3. 브라우저 위치 권한을 요청한다.
4. GPS 좌표가 등록 근무지 반경 안인지 확인한다.
5. 성공 시 출근 기록을 생성한다.
6. 실패 시 QR 인증 또는 Wi-Fi/IP 보조 인증을 제안한다.
7. 모든 인증 실패 시 정정 요청을 생성할 수 있다.

### 3.2 직원 퇴근

1. 직원이 `퇴근` 버튼을 누른다.
2. GPS 또는 보조 인증으로 근무지 검증을 수행한다.
3. 퇴근 시간이 17:00 이전이면 조기퇴근 시간을 계산한다.
4. 퇴근 기록과 조기퇴근 데이터를 저장한다.
5. 직원은 당월 조기퇴근 누계를 확인한다.

### 3.3 휴가/연차/반차 신청

1. 직원이 휴가 신청 화면을 연다.
2. 사용 가능한 잔여 법정 연차와 회사 선사용 가능 한도를 본다.
3. 연차, 반차, 일반 휴가 유형을 선택한다.
4. 날짜, 사유, 대체 인력 메모를 입력한다.
5. 시스템이 잔여 한도와 중복 신청을 검증한다.
6. 승인자에게 신청이 전달된다.
7. 승인 시 휴가 장부에 반영된다.

### 3.4 입사 첫해 선사용 휴가

1. 입사 3개월차부터 회사 정책상 선사용 가능 상태가 된다.
2. 법정 연차가 아직 발생하지 않았더라도 회사 선사용 휴가 장부에서 차감한다.
3. 다음해 법정 연차 발생 시 선사용분을 상계한다.
4. 퇴사 또는 미발생 상황이 생기면 HR 관리자가 정산 상태를 확인한다.

### 3.5 야근 신청

1. 직원이 야근 신청을 등록한다.
2. 날짜, 예상 시작/종료 시간, 사유를 입력한다.
3. 시스템이 주간 연장근로 누계를 계산한다.
4. 1주 12시간 초과 가능성이 있으면 경고한다.
5. 승인자가 승인/반려한다.
6. 실제 퇴근 기록과 신청 시간을 비교해 관리자 화면에 표시한다.

## 4. 출퇴근 인증 정책

### 4.1 인증 우선순위

| 순위 | 방식 | 용도 |
|---|---|---|
| 1 | GPS | 기본 출퇴근 인증 |
| 2 | QR | GPS 실패 또는 실내 오차 보완 |
| 3 | Wi-Fi/IP | 보조 신호, 신뢰도 보강 |
| 4 | GPS수신실패 허용 | GPS 오류 시 QR 또는 버튼 클릭으로 출퇴근 허용 |
| 5 | 정정/보정 | 예외 상황 승인 및 관리자 시간 보정 |

### 4.2 GPS 정책

- 직원이 출근/퇴근 버튼을 누르는 순간에만 GPS를 요청한다.
- 상시 위치 추적은 하지 않는다.
- 근무지별 중심 좌표와 허용 반경을 설정한다.
- 회사 2개 사무실은 기본 반경 300m로 설정한다.
- GPS 수신 실패, 권한 오류, 정확도 부족 등 문제가 있으면 직원에게 `GPS수신실패`를 노티한다.
- GPS 실패 상태에서도 직원이 QR을 스캔하거나 출근/퇴근 버튼을 클릭하면 출퇴근은 허용한다.
- 이 경우 출퇴근 기록에는 `GPS_FAILED_ALLOWED` 상태를 남긴다.
- 저장 데이터는 최소화한다.
  - 필수: 인증 결과, 인증 방식, 시각, 근무지 ID, 정확도
  - 선택: 원시 좌표
- 원시 좌표를 저장한다면 보관 기간을 짧게 설정한다.

### 4.3 QR 정책

- MVP에서는 출입문에 고정 QR 1개를 출력해 부착한다.
- 고정 QR은 비밀 토큰이 아니라 해당 사무실 체크인 URL 또는 `workplaceId`를 담는다.
- QR 인증은 GPS 실패 시 출퇴근 허용을 보조하고, 기록에는 `GPS수신실패+QR`로 남긴다.
- 고정 QR은 공유 가능성이 있으므로 관리자 화면에서 GPS 실패 QR 출퇴근 내역을 필터링할 수 있어야 한다.

### 4.4 Wi-Fi/IP 정책

- Wi-Fi는 불안정하므로 메인 인증이 아닌 보조 신호로 사용한다.
- 회사 고정 IP 또는 내부망 접근 여부를 확인한다.
- 모바일 통신망 전환, 공유기 문제 등으로 실패해도 출퇴근 자체를 막지 않는다.

### 4.5 정정 요청

- 위치 권한 거부, GPS 오류, QR 미작동, 긴급 외근 등은 정정 요청으로 처리한다.
- 정정 요청에는 사유와 희망 출퇴근 시각을 입력한다.
- 승인 시 원본 요청과 승인자, 승인 시각을 감사 로그에 남긴다.

### 4.6 관리자 출퇴근 보정

- 관리자는 개인별 출퇴근 시간을 보정할 수 있다.
- 보정 유형은 인정지각, 인정조퇴, 출근시각 보정, 퇴근시각 보정, 누락 기록 추가로 둔다.
- 원본 기록은 삭제하지 않고 보정 전/후 값을 모두 남긴다.
- 보정 사유 입력은 필수다.
- 보정자, 보정 시각, 대상 직원, 원본 시각, 변경 시각, 사유를 감사 로그에 남긴다.

## 5. 근무시간 및 조기퇴근 정책

### 5.1 기본 근무시간

- 기본 출근: 08:00
- 기본 퇴근: 17:00
- 기본 근무일: 회사 정책으로 설정한다.
- 업무 강도에 따라 유연 운영하되, 시스템은 실제 출퇴근 기록과 조기퇴근 시간을 투명하게 보여준다.

### 5.2 조기퇴근 계산

- 퇴근 시간이 17:00 이전이면 `17:00 - 실제 퇴근 시각`을 조기퇴근 시간으로 계산한다.
- 휴가/반차/승인된 조기퇴근/업무강도에 따른 유연퇴근 사유가 있으면 별도 구분한다.
- 단순 누계만 제공하고, 급여 차감 여부는 MVP에서 자동 처리하지 않는다.
- 성수기인 1월, 2월은 조기퇴근-야근 상계에서 제외한다.
- 성수기를 제외한 평달 야근은 기본적으로 초과근무수당을 자동 계산하지 않고 조기퇴근 누적분에서 먼저 차감한다.
- 단, 관리자가 인정한 초과근무 시간은 수당 집계 대상으로 표시한다.

### 5.3 조기퇴근 상태

| 상태 | 설명 |
|---|---|
| APPROVED | 사전 승인된 조기퇴근 |
| FLEX_ALLOWED | 업무 강도, 현장 상황 등으로 관리자가 유연 운영 처리 |
| LEAVE_RELATED | 반차, 휴가 등 승인된 휴가와 연결 |
| UNAPPROVED | 별도 승인 없이 17:00 이전 퇴근 |
| CORRECTED | 정정 요청으로 조정된 기록 |

### 5.4 조기퇴근 메뉴

직원 화면:

- 이번 달 조기퇴근 누계
- 기간별 조기퇴근 내역
- 사유/승인 상태
- 본인 데이터 다운로드

관리자 화면:

- 직원별 월별 조기퇴근 시간
- 부서별 기간별 요약
- 승인된 조기퇴근과 미승인 조기퇴근 구분
- 유연 운영 처리된 조기퇴근 구분
- CSV 다운로드

## 6. 휴가·연차 정책

### 6.1 장부 분리

법정 연차와 회사 선사용 휴가는 반드시 분리한다.

| 장부 | 설명 |
|---|---|
| 법정 연차 장부 | 근로기준법 기준으로 발생·사용·소멸 관리 |
| 선사용 휴가 장부 | 회사가 입사 첫해에 유연하게 허용하는 선사용분 관리 |
| 조정 장부 | 퇴사, 오류 수정, 특별 승인 등 수동 조정 관리 |

### 6.2 법정 연차 기본 규칙

- 1년간 80% 이상 출근한 근로자에게 15일 부여
- 계속 근로 1년 미만 또는 1년간 80% 미만 출근 시 1개월 개근마다 1일 부여
- 3년 이상 계속 근로 시 최초 1년 초과 후 매 2년에 1일 가산
- 총 휴가 일수는 25일 한도
- 법정 기준은 정책 모듈에서 설정값으로 관리한다.

### 6.3 입사 첫해 선사용 휴가

회사 정책:

- 입사 3개월차부터 선사용 휴가 신청 가능
- 선사용 가능 한도는 관리자 설정값으로 둔다.
- 선사용 휴가 총 한도는 다음해까지 사용할 수 있게 지급될 법정 연차 총 개수로 제한한다.
- 입사 첫해 월별로 발생하는 법정 연차가 있으면 법정 장부를 우선 차감하고, 부족분만 선사용 장부에서 차감한다.
- 선사용분은 다음 법정 연차 발생 시 자동 상계한다.
- 퇴사 시 미상계 선사용분은 HR이 정산 상태를 확인할 수 있어야 한다.

중요 설계:

- 선사용 휴가는 법정 발생 연차로 표시하지 않는다.
- 직원 화면에는 `사용 가능 휴가`로 합산 표시하되, 상세에는 법정/선사용을 구분한다.
- 관리자 화면에서는 법정 연차, 선사용, 상계 예정, 미상계 잔액을 구분한다.

### 6.4 반차/휴가 단위

- 연차: 1.0일
- 반차: 0.5일
- 일반 휴가: 유급/무급/특별휴가 유형으로 설정 가능
- 시간차는 MVP에서 제외하고, 추후 확장 항목으로 둔다.

## 7. 급여명세서

### 7.1 MVP 방식

- HR 관리자가 월별 급여명세서 PDF 또는 이미지 파일을 업로드한다.
- 직원은 본인 명세서만 조회한다.
- 열람, 다운로드 이력을 남긴다.
- 급여 자동 계산은 MVP 범위에서 제외한다.

### 7.2 확장 가능성

- 급여대장 CSV 업로드
- 외부 급여 시스템 연동
- 세전/세후 요약 카드
- 전자문서 교부 확인

## 8. 비즈니스 로직 모듈 구조

### 8.1 모듈 개요

| 모듈 | 책임 |
|---|---|
| Auth & RBAC | 로그인, 세션, 역할 기반 접근 |
| Employee | 직원, 부서, 입사일, 근무지 |
| Workplace | GPS 반경, QR, IP 설정 |
| Attendance | 출근/퇴근, 인증 결과, 정정 요청 |
| Verification | GPS/QR/Wi-Fi 인증 판정 |
| WorkPolicy | 기본 근무시간, 유연 운영 기준 |
| LeavePolicy | 법정 연차, 선사용 정책 계산 |
| LeaveRequest | 휴가/연차/반차 신청과 승인 |
| EarlyLeave | 조기퇴근 계산과 조회 |
| Overtime | 야근 신청, 승인, 주간 한도 경고 |
| Payroll | 급여명세서 업로드와 조회 |
| Notification | 승인 요청, 결과 알림 |
| AuditLog | 민감 작업 및 변경 이력 |

### 8.2 모듈화 원칙

- UI는 도메인 계산을 직접 수행하지 않는다.
- 각 도메인은 `policy`, `service`, `repository`로 나눈다.
- 정책 계산은 순수 함수에 가깝게 유지한다.
- DB 스키마와 API는 단순하게 시작한다.
- 이벤트 기반 구조는 초기에 과하게 도입하지 않는다.
- 단, 감사 로그와 알림은 서비스 레벨 이벤트처럼 호출할 수 있게 경계를 둔다.

## 13. 수용 기준

### 13.1 출퇴근

- 직원은 모바일에서 GPS로 출근/퇴근할 수 있다.
- GPS 실패 시 QR 인증을 시도할 수 있다.
- Wi-Fi/IP는 보조 판정으로 기록된다.
- 인증 실패 시 정정 요청을 생성할 수 있다.
- 17:00 이전 퇴근 시 조기퇴근 시간이 자동 계산된다.
- 조기퇴근은 승인, 유연운영, 휴가연동, 미승인, 정정 상태로 구분된다.

### 13.2 휴가

- 입사일 기준 법정 연차가 계산된다.
- 입사 3개월차부터 선사용 휴가 신청이 가능하다.
- 법정 연차와 선사용 휴가가 장부상 분리된다.
- 승인된 휴가만 잔여 휴가에 반영된다.
- 반차는 0.5일로 처리된다.

### 13.3 관리자

- 관리자는 직원별 월별 조기퇴근 시간을 조회할 수 있다.
- 관리자는 기간별 조기퇴근 데이터를 필터링할 수 있다.
- 관리자는 휴가/야근/정정 요청을 승인 또는 반려할 수 있다.
- 관리자는 급여명세서를 직원별 월별로 업로드할 수 있다.

## 15. 서브에이전트 오케스트레이션 계획

### 15.1 운영 방식

메인 에이전트는 전체 아키텍처, 인터페이스 계약, PR 병합 기준을 관리한다. 서브에이전트는 도메인별로 병렬 작업하되, 각 작업은 작은 PR 단위로 제출한다.

공통 규칙:

- 각 서브에이전트는 자기 도메인 밖의 파일 수정을 최소화한다.
- DB 모델 변경은 메인 에이전트 승인 후 진행한다.
- 비즈니스 로직은 단위 테스트를 포함한다.
- UI는 모바일 뷰를 먼저 검증한다.
- 개인정보/급여/위치 관련 변경은 보안 리뷰 에이전트가 확인한다.

### 15.2 서브에이전트 역할

| 에이전트 | 책임 | 주요 산출물 |
|---|---|---|

| Architecture Agent | 모듈 구조, DB/API 계약 | 도메인 경계, API 스펙, 스키마 |

### 15.3 병렬 개발 순서

1. Architecture Agent가 공통 모델과 API 계약 초안을 만든다.
2. Product Agent가 직원/관리자 화면 흐름을 확정한다.
3. Attendance, Leave, EarlyLeave Agent가 각 도메인 로직을 병렬 구현한다.
4. Frontend Agent가 API mock 기반으로 UI를 만든다.
5. Overtime, Payroll Agent가 독립 기능을 붙인다.
6. Security Agent가 위치/급여/권한/로그를 리뷰한다.
7. QA Agent가 모바일 중심 E2E를 검증한다.

## 18. 최종 권장안

MVP는 `GPS 우선 + QR 보조 + Wi-Fi 참고 + 정정 승인`으로 시작한다.  
휴가 관리는 `법정 연차 장부`와 `회사 선사용 휴가 장부`를 분리한다.  
조기퇴근은 자동 계산하되, 급여 차감이나 평가 반영은 시스템에 바로 넣지 않고 조회·승인·기록 기능부터 제공한다.  
이 구조가 가장 단순하면서도 추후 교대근무, 지점, 재택근무, 급여시스템 연동으로 확장하기 좋다.
