# CTO 보고: 사내 근태·휴가·급여 확인 웹서비스 다음 진행안

작성일: 2026-07-08  
기반 문서: `internal-hr-attendance-prd.md`

## 1. 현재 정리된 방향

사내 직원용 웹서비스는 모바일 우선으로 구축한다. 직원은 휴대폰에서 출퇴근, 휴가/연차/반차 신청, 야근 신청, 급여명세서 확인을 처리하고, 관리자는 승인·조회·정정·급여명세서 업로드를 수행한다.

출퇴근 인증은 GPS를 메인으로 사용한다. Wi-Fi는 불안정하므로 보조 신호로만 사용하고, QR을 GPS 실패 또는 실내 오차 대응 수단으로 적용한다. 모든 인증 실패 상황은 정정 요청과 관리자 승인으로 처리한다.

휴가 관리는 법정 연차와 회사 선사용 휴가를 분리한다. 입사 첫해에는 3개월차부터 다음해 발생 예정 연차를 선사용할 수 있도록 하되, 법정 연차 장부와 선사용 장부를 분리해 정산 리스크를 낮춘다.

근무시간은 08:00 출근, 17:00 퇴근을 기본으로 한다. 17:00 이전 퇴근 시 조기퇴근 시간을 자동 계산하고, 월별/기간별 데이터를 직원과 관리자가 조회할 수 있게 한다. 업무 강도에 따른 유연 운영은 `FLEX_ALLOWED` 상태로 별도 구분한다.

## 2. CTO 관점 핵심 판단

### 2.1 기술 방향

- MVP는 단일 웹 애플리케이션으로 시작한다.
- 모바일 웹/PWA 수준으로 구현하고, 네이티브 앱은 후순위로 둔다.
- 비즈니스 로직은 `Attendance`, `Verification`, `Leave`, `EarlyLeave`, `Overtime`, `Payroll`, `Audit` 모듈로 분리한다.
- 이벤트 기반, 마이크로서비스, 복잡한 전자결재 엔진은 초기 범위에서 제외한다.
- 정책값은 코드에 박지 않고 관리자 설정 또는 정책 테이블로 분리한다.

### 2.2 제품 방향

- 첫 화면은 실제 업무 화면이어야 한다.
- 직원은 출근/퇴근, 휴가 신청, 조기퇴근 누계 확인을 빠르게 처리해야 한다.
- 관리자는 승인 대기, 이상 근태, 조기퇴근 누계, 급여명세서 업로드를 한 화면 흐름으로 처리해야 한다.
- HR 운영자가 수동 보정할 수 있는 관리자 기능이 반드시 필요하다.

### 2.3 리스크 판단

- GPS는 편하지만 실내 오차와 위치정보 민감도가 있다.
- QR은 보조 인증으로 유용하지만 공유 가능성이 있다.
- 선사용 휴가는 직원 경험에는 좋지만 퇴사/정산 리스크가 있다.
- 조기퇴근 데이터는 인사평가나 급여 차감으로 연결될 수 있어 접근권한과 해석 기준이 필요하다.
- 급여명세서는 민감정보이므로 접근 로그와 권한 분리가 필수다.

## 3. 바로 결정해야 할 사항

| 결정 항목 | 권장안 | 결정 필요 이유 |
|---|---|---|
| GPS 원시 좌표 저장 여부 | 기본은 저장하지 않고 판정값만 저장 | 개인정보 리스크 최소화 |
| GPS 허용 반경 | 기본 100m, 근무지별 조정 | 실내 오차와 부정확성 대응 |
| QR 유효시간 | 1~5분 단위 토큰 | QR 공유 리스크 완화 |
| 선사용 휴가 한도 | 최대 5일 또는 예상 발생 연차 내 | 퇴사 정산 리스크 통제 |
| 조기퇴근 처리 기준 | 승인/유연운영/휴가연동/미승인/정정 구분 | 데이터 해석 혼선 방지 |
| 급여명세서 보관 기간 | 내부 규정에 따라 확정 | 민감정보 관리 필요 |
| 승인자 구조 | 직원별 1차 승인자 지정 | 조직도 없이 MVP 가능 |

## 4. 다음 실행 순서

### Step 1: 아키텍처와 데이터 모델 확정

기간: 2~3일  
담당: Architecture Agent, CTO 리뷰

산출물:

- 도메인 모듈 경계
- DB 스키마 초안
- API 계약
- 권한 매트릭스

완료 기준:

- 직원, 근무지, 출퇴근, 인증, 휴가 장부, 조기퇴근, 야근, 급여명세서 모델이 정의된다.
- 법정 연차와 선사용 휴가 장부가 분리되어 있다.

### Step 2: 정책 결정 워크숍

기간: 1일  
담당: CTO, HR, 운영 책임자

결정할 내용:

- 선사용 휴가 한도
- GPS 좌표 보관 여부
- QR 운영 방식
- 조기퇴근 데이터의 용도
- 정정 요청 승인 기준
- 급여명세서 보관/열람 정책

완료 기준:

- 개발자가 임의 해석 없이 구현할 수 있는 정책값이 확정된다.

### Step 3: MVP 개발 착수

기간: 2~4주  
담당: 도메인별 서브에이전트

우선순위:

1. 인증/권한/직원/근무지
2. GPS/QR/Wi-Fi 출퇴근
3. 휴가 장부와 신청/승인
4. 조기퇴근 계산과 조회
5. 야근 신청/승인
6. 급여명세서 업로드/조회
7. 감사 로그와 개인정보 고지

완료 기준:

- 직원이 모바일에서 출근/퇴근, 휴가 신청, 야근 신청, 급여명세서 조회를 할 수 있다.
- 관리자가 승인, 정정, 조기퇴근 조회, 급여명세서 업로드를 할 수 있다.

### Step 4: 보안·개인정보 리뷰

기간: 2~3일  
담당: Security Agent, CTO 리뷰

점검 항목:

- 위치정보 최소 수집
- 급여명세서 접근권한
- 관리자 권한 변경 로그
- 다운로드/열람 로그
- 정정 요청 감사 로그
- 개인정보 고지 문구

완료 기준:

- 위치정보, 급여명세서, 조기퇴근 데이터 접근 이력이 추적된다.
- 직원에게 필요한 고지 문구가 서비스 내에 포함된다.

### Step 5: 파일럿 운영

기간: 1~2주  
대상: 일부 부서 또는 5~10명

검증 항목:

- GPS 인증 성공률
- QR 보조 인증 사용 빈도
- 정정 요청 발생 원인
- 연차/선사용 휴가 계산 정확도
- 조기퇴근 데이터 해석 혼선 여부
- 관리자 승인 업무 부담

완료 기준:

- 운영자가 매일 사용할 수 있을 정도로 오류와 예외 흐름이 정리된다.

## 5. 서브에이전트 오케스트레이션 제안

메인 에이전트는 전체 아키텍처, API 계약, PR 병합 기준을 관리한다. 서브에이전트는 도메인별로 병렬 개발한다.

| 에이전트 | 우선순위 | 주요 책임 |
|---|---:|---|
| Architecture Agent | 1 | 스키마, API, 모듈 경계 |
| Product Agent | 1 | 화면 흐름, 수용 기준 |
| Attendance Agent | 2 | GPS/QR/Wi-Fi 출퇴근 |
| Leave Agent | 2 | 법정 연차, 선사용 휴가 |
| EarlyLeave Agent | 2 | 조기퇴근 계산/조회 |
| Frontend Agent | 3 | 직원 모바일 UI, 관리자 UI |
| Security Agent | 3 | 권한, 위치정보, 급여명세서 리뷰 |
| Overtime Agent | 4 | 야근 신청/승인 |
| Payroll Agent | 4 | 급여명세서 업로드/조회 |
| QA Agent | 5 | E2E, 회귀 테스트 |

## 6. 추천 개발 원칙

- 복잡한 근태 시스템을 처음부터 만들지 않는다.
- 법정 연차와 회사 복지성 선사용 휴가를 데이터 구조에서 분리한다.
- 출퇴근 인증은 `인증 시도`와 `근태 기록`을 분리해 저장한다.
- 조기퇴근은 급여 차감 로직과 연결하지 않고, 조회·상태·승인 데이터로 시작한다.
- 모든 민감 데이터는 접근 로그를 남긴다.
- 관리자가 수동 정정할 수 있는 기능을 MVP에 포함한다.

## 7. CTO에게 요청되는 의사결정

1. MVP 기술 스택 확정
2. GPS 원시 좌표 저장 여부 확정
3. 선사용 휴가 한도 확정
4. 조기퇴근 데이터의 활용 범위 확정
5. 급여명세서 보관/열람 정책 확정
6. 파일럿 대상 부서 확정

## 8. 결론

다음 단계는 바로 개발 착수가 아니라, 2~3일 안에 아키텍처·데이터 모델·정책값을 확정하는 것이다. 이후 도메인별 서브에이전트를 병렬 투입하면 MVP를 2~4주 단위로 만들 수 있다.

가장 중요한 설계 원칙은 세 가지다.

- 출퇴근은 `GPS 우선 + QR 보조 + Wi-Fi 참고 + 정정 승인`
- 휴가는 `법정 연차 장부 + 선사용 휴가 장부 분리`
- 조기퇴근은 `자동 계산 + 상태 구분 + 조회 중심`

이 방향이면 초기 구현은 단순하게 유지하면서도, 추후 지점근무, 재택근무, 교대근무, 급여시스템 연동으로 확장할 수 있다.
