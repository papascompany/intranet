# 대표 결정사항 반영 CTO 구현 메모

작성일: 2026-07-08  
역할 가정: 대표 의사결정 반영, CTO 구현 정책 정리

## 1. 대표 결정사항 요약

| 항목 | 대표 결정 | CTO 반영 방향 |
|---|---|---|
| GPS | 회사 2개 사무실 좌표 기준 반경 300m 이내 허용 | 근무지별 좌표와 반경 300m를 정책값으로 저장 |
| QR | 고정 QR 1개를 출력해 출입문에 부착 | 고정 QR은 단독 인증이 아니라 보조 인증으로 사용 |
| 선사용 휴가 | 입사 3개월 이후 매월 1개씩 충전, 다음해 지급 연차 총 개수 한도 | 입사 3개월 경과 후 매월 1일씩 적립하되 다음해 발생 예정 연차 총량을 상한으로 설정 |
| 조기퇴근 상계 | 평달 야근 시 수당 대신 조기퇴근 누적분에서 차감, 관리자 인정분만 수당 집계 | 1월/2월 제외, 일반 야근은 조기퇴근 누적분과 상계, 관리자 인정 야근만 수당 집계 |
| 급여명세서 | 관리자가 삭제하기 전까지 지속 보관 | 관리자 삭제 전까지 보관, 삭제 로그 필수 |
| 파일럿 | 운영팀, 대표 포함 4명 | 운영팀 우선 적용 후 제작팀 확대 |
| 출퇴근 보정 | 잘못된 출퇴근 기록은 관리자가 개인별 보정 | 인정지각, 인정조퇴, 시간 보정, 사유 기록과 감사 로그 구현 |

## 2. CTO 최종 권장안

대표 결정사항은 MVP 정책으로 반영한다. QR은 고정 출력 방식을 유지하되, 출퇴근 단독 증빙으로 과신하지 않는다. GPS가 정상 수신되고 300m 반경 안이면 정상 출퇴근으로 처리한다. GPS 수신 실패 또는 오류가 있으면 `GPS수신실패`로 노티/마킹하고, 직원이 QR을 스캔하거나 출근/퇴근 버튼을 다시 클릭하면 출퇴근은 허용하되 인증 상태를 별도로 남긴다.

이렇게 하면 운영은 단순하게 유지하면서도 GPS 실패로 인한 현장 불편을 줄일 수 있다. 다만 GPS 실패 출퇴근은 관리자 화면에서 별도 필터링되도록 만들어야 한다.

## 3. 정책 상세

### 3.1 GPS 출퇴근

정책:

- 사무실은 2개를 등록한다.
- 각 사무실은 `latitude`, `longitude`, `allowedRadiusMeters`를 가진다.
- MVP 기본 반경은 두 사무실 모두 300m로 설정한다.
- 직원이 출근/퇴근 버튼을 누르는 순간 GPS를 1회 요청한다.
- 현재 위치가 두 사무실 중 하나의 300m 반경 안이면 GPS 인증을 통과한다.
- GPS 수신 실패, 권한 오류, 정확도 부족 등 문제가 있으면 직원에게 `GPS수신실패`를 노티한다.
- GPS 실패 상태에서도 직원이 QR을 스캔하거나 출근/퇴근 버튼을 클릭하면 출퇴근은 허용한다.
- 이 경우 출퇴근 기록에는 `GPS_FAILED_ALLOWED` 상태를 남긴다.

저장 권장값:

- 직원 ID
- 출퇴근 시각
- 인증 결과
- 매칭된 사무실 ID
- 거리값
- GPS 정확도
- 원시 좌표 저장 여부

CTO 의견:

- 300m는 운영 편의성은 좋지만 도심에서는 카페, 인근 건물까지 포함될 수 있다.
- 파일럿 기간에는 300m로 시작하고, 실제 오탐이 많으면 150m 또는 100m로 줄이는 방식이 좋다.

### 3.2 QR 인증

대표안:

- QR 1개를 출력해 출입문에 붙인다.

CTO 보완안:

- 고정 QR은 가능하다.
- 단, QR 안에는 비밀 토큰을 넣지 않는다.
- QR은 `해당 사무실 체크인 URL` 또는 `workplaceId`만 담는다.
- QR 스캔 후 서버는 로그인 사용자, 현재 시각, GPS 결과, 사무실 정보, 중복 출퇴근 여부를 검증한다.
- QR은 다음 중 하나로 처리한다.
  - GPS 300m 성공 + QR 스캔: 정상 출퇴근
  - GPS 실패 + QR 스캔: 출퇴근 허용, `GPS수신실패+QR`로 마킹
  - GPS 없음 + QR 스캔: 출퇴근 허용, `GPS수신실패+QR`로 마킹
  - GPS 실패 + 버튼 클릭: 출퇴근 허용, `GPS수신실패+수동클릭`으로 마킹

이유:

- 고정 QR은 사진으로 공유될 수 있다.
- QR을 비밀키처럼 쓰면 한 번 유출된 순간 계속 위험하다.
- 따라서 고정 QR은 운영 편의용으로 쓰고, GPS 실패 마킹과 관리자 필터로 사후 확인 가능하게 만든다.

### 3.3 선사용 휴가 충전

정책:

- 입사 후 3개월이 지나면 선사용 휴가 적립 대상이 된다.
- 입사 3개월 경과 시점부터 매월 1일씩 선사용 휴가가 충전된다.
- 선사용 휴가 총 한도는 다음해까지 사용할 수 있게 지급될 법정 연차 총 개수로 제한한다.
- 충전 단위는 1일이다.
- 반차 사용 시 0.5일 차감한다.
- 선사용 휴가는 법정 연차 장부와 분리한다.
- 다음해 법정 연차 발생 시 선사용분을 상계한다.

예시:

- 입사일: 2026-01-10
- 3개월 경과: 2026-04-10
- 2026-04-10: 선사용 휴가 1일 충전
- 2026-05-10: 선사용 휴가 1일 추가 충전, 누적 2일
- 2026-06-10: 선사용 휴가 1일 추가 충전, 누적 3일

구현 규칙:

- `AdvanceLeaveLedger`에 충전 내역을 월별로 기록한다.
- `AdvanceLeavePolicy`는 직원별 다음해 발생 예정 연차 총량을 계산하고, 선사용 누적 충전량이 이를 초과하지 못하게 한다.
- 사용 시 가장 먼저 법정 연차 사용 가능분을 차감하고, 부족분만 선사용 휴가에서 차감한다.
- 퇴사 시 미상계 선사용 휴가를 HR 관리자 화면에 표시한다.

확인 필요:

- 결근, 휴직, 수습 중단 등 예외 상황에서 충전을 멈출지 여부

CTO 권장:

- 대표 결정에 따라 총 한도는 다음해 지급 연차 총량으로 확정한다.
- MVP에서는 휴직/장기 결근 같은 예외가 발생하면 자동 계산보다 HR 관리자 보정으로 처리한다.

### 3.4 조기퇴근 누적분과 야근 상계

정책:

- 기본 퇴근 시간은 17:00이다.
- 17:00 이전 퇴근 시 조기퇴근 시간이 누적된다.
- 성수기인 1월, 2월은 상계 정책에서 제외한다.
- 성수기를 제외한 평달에 야근이 발생하면, 기본적으로 초과근무수당을 계산하지 않고 조기퇴근 누적분에서 먼저 차감한다.
- 단, 관리자가 초과근무를 인정한 경우에는 해당 초과근무 시간만 수당 집계 대상으로 표시한다.

계산 예시:

- 직원 A의 조기퇴근 누적: 180분
- 3월 야근 승인 시간: 60분
- 수당 계산 대상: 0분
- 조기퇴근 잔여 누적: 120분

추가 예시:

- 직원 A의 조기퇴근 누적: 30분
- 3월 야근 승인 시간: 90분
- 조기퇴근 상계: 30분
- 잔여 야근 시간: 60분
- 잔여 야근 시간은 자동 수당 계산하지 않는다.
- 관리자가 인정한 잔여 야근 시간만 `수당 집계 대상`으로 표시한다.

상태값:

- EARLY_LEAVE_ACCRUED: 조기퇴근 누적
- OVERTIME_APPROVED: 승인된 야근
- OFFSET_APPLIED: 야근과 조기퇴근 상계 완료
- OFFSET_EXCLUDED_PEAK_SEASON: 1월/2월 성수기로 상계 제외
- OVERTIME_PAY_APPROVED: 관리자가 초과근무수당 집계를 인정
- OVERTIME_PAY_NOT_COUNTED: 관리자가 인정하지 않아 수당 미집계

CTO 의견:

- 이 정책은 근태/급여에 영향을 줄 수 있으므로 직원에게 명확히 고지해야 한다.
- MVP에서는 급여 자동 반영보다 `상계 장부`, `관리자 인정`, `수당 집계 대상 표시`까지 구현하는 것이 안전하다.

### 3.5 출퇴근 보정

정책:

- 출퇴근 기록이 잘못된 경우 관리자가 개인별 출근/퇴근 시간을 보정할 수 있다.
- 보정 유형은 인정지각, 인정조퇴, 출근시각 보정, 퇴근시각 보정, 누락 기록 추가로 둔다.
- 보정 시 원본 기록은 삭제하지 않고, 보정 전/후 값을 모두 남긴다.
- 보정에는 사유 입력을 필수로 한다.
- 보정자, 보정 시각, 대상 직원, 원본 시각, 변경 시각, 사유를 감사 로그에 남긴다.

보정 유형:

- APPROVED_LATE: 인정지각
- APPROVED_EARLY_LEAVE: 인정조퇴
- CLOCK_IN_CORRECTION: 출근시각 보정
- CLOCK_OUT_CORRECTION: 퇴근시각 보정
- MISSING_RECORD_CREATED: 누락 출퇴근 기록 생성

CTO 의견:

- 이 기능은 MVP 필수다. GPS 실패 허용 정책을 두는 만큼, 관리자 보정과 감사 로그가 있어야 운영이 안정된다.

### 3.6 급여명세서 보관

정책:

- 급여명세서는 관리자가 삭제하기 전까지 지속 보관한다.
- 직원은 본인 명세서만 조회한다.
- HR 관리자와 시스템 관리자는 권한에 따라 조회/업로드/삭제할 수 있다.
- 삭제 시 삭제자, 삭제 시각, 대상 파일, 사유를 감사 로그에 남긴다.

CTO 보완:

- 실제 파일은 직접 공개 URL로 노출하지 않는다.
- 다운로드는 권한 확인 후 일회성 링크 또는 서버 스트리밍으로 제공한다.
- 삭제는 물리 삭제보다 `soft delete`로 시작하는 것이 좋다.

### 3.7 파일럿 범위

정책:

- 파일럿 대상은 운영팀이다.
- 운영팀 인원은 대표 포함 4명이다.
- 제작팀 5명은 파일럿 안정화 후 적용한다.

파일럿 체크 항목:

- GPS 300m 인증 성공률
- GPS수신실패 마킹 빈도
- QR 보조 인증 사용 빈도
- 정정 요청 발생 이유
- 관리자 출퇴근 보정 건수
- 선사용 휴가 충전/차감 정확도
- 조기퇴근-야근 상계 결과에 대한 직원 이해도
- 관리자 승인 업무량

## 4. 개발 반영 사항

### 4.1 데이터 모델 추가/수정

필요 모델:

- `Workplace`: 2개 사무실 좌표와 반경 300m
- `VerificationAttempt`: GPS, QR, 보조 인증 결과
- `AdvanceLeaveLedger`: 선사용 휴가 월별 충전/사용/상계
- `EarlyLeaveLedger`: 조기퇴근 누적
- `OvertimeOffsetLedger`: 야근과 조기퇴근 상계 내역
- `AttendanceCorrection`: 관리자 출퇴근 보정 내역
- `PayrollStatement`: 급여명세서 파일 메타데이터
- `AuditLog`: 급여명세서 삭제, 위치 인증, 정정 승인, 출퇴근 보정 이력

### 4.2 관리자 설정 화면

MVP에 필요한 설정:

- 사무실 2개 좌표 등록
- GPS 허용 반경 300m
- QR 출력용 URL 생성
- 선사용 휴가 충전 시작 기준: 입사 3개월 경과
- 선사용 휴가 월 충전량: 1일
- 선사용 휴가 총 한도: 다음해 지급 예정 연차 총 개수
- 성수기 월: 1월, 2월
- 급여명세서 보관 정책: 관리자 삭제 전까지 보관

### 4.3 직원 화면

필요 화면:

- 출근/퇴근
- QR 스캔
- 휴가 잔액
- 선사용 휴가 잔액
- 조기퇴근 누적 시간
- 야근 상계 내역
- GPS수신실패로 기록된 출퇴근 내역
- 급여명세서

### 4.4 관리자 화면

필요 화면:

- 운영팀 파일럿 직원 목록
- 출퇴근 인증 내역
- 정정 요청 승인
- 개인별 출퇴근 시간 보정
- 인정지각/인정조퇴 처리
- 선사용 휴가 장부
- 조기퇴근 누적/상계 내역
- 야근 승인 및 상계 결과
- 급여명세서 업로드/삭제
- 감사 로그

## 5. 서브에이전트 작업 지시 업데이트

### Architecture Agent

- 대표 결정사항을 반영해 DB 스키마를 수정한다.
- `OvertimeOffsetLedger`와 `AdvanceLeaveLedger`를 별도 모델로 둔다.
- QR은 고정 URL 기반으로 설계하되, 단독 인증으로 처리하지 않는다.

### Attendance Agent

- GPS 300m 반경 판정 로직을 구현한다.
- 2개 사무실 중 어느 곳에 매칭됐는지 기록한다.
- 고정 QR 스캔은 보조 인증 이벤트로 기록한다.
- GPS 실패 시 `GPS수신실패` 노티와 기록 마킹을 구현한다.
- GPS 실패 + QR 또는 버튼 클릭 시 출퇴근은 허용하되 인증 상태를 별도 저장한다.
- 관리자 출퇴근 보정 기능과 감사 로그를 구현한다.

### Leave Agent

- 입사 3개월 경과 후 매월 1일 선사용 휴가 충전 로직을 구현한다.
- 선사용 휴가 총 한도를 다음해 지급 예정 연차 총 개수로 제한한다.
- 선사용 휴가는 별도 장부에 기록한다.
- 법정 연차 발생 시 상계 로직을 준비한다.

### EarlyLeave Agent

- 17:00 이전 퇴근 시 조기퇴근 누적 장부에 기록한다.
- 성수기 제외 월의 야근 승인분과 조기퇴근 누적분을 상계한다.
- 초과근무수당은 자동 계산하지 않고, 관리자가 인정한 초과근무 시간만 수당 집계 대상으로 표시한다.
- 상계 결과를 직원/관리자 화면에 노출할 수 있게 API를 만든다.

### Payroll Agent

- 급여명세서를 관리자 삭제 전까지 보관한다.
- 삭제는 soft delete로 구현한다.
- 조회, 다운로드, 삭제 감사 로그를 남긴다.

### Security Agent

- 고정 QR의 오남용 리스크를 검토한다.
- GPS 좌표 저장 범위와 보관 정책을 확인한다.
- GPS수신실패 상태로 허용된 출퇴근의 감사 가능성을 검토한다.
- 관리자 출퇴근 보정 권한과 로그를 검토한다.
- 급여명세서 접근권한과 삭제 로그를 검토한다.
- 조기퇴근-야근 상계 정책 고지 필요 여부를 점검한다.

## 6. 남은 의사결정

대표 결정사항 대부분은 구현 가능하다. 아래 항목만 운영 전 추가 확정이 필요하다.

| 항목 | 질문 | CTO 권장 |
|---|---|---|
| GPS 원시 좌표 저장 | 좌표 자체를 보관할 것인가 | 기본은 판정값, 거리, 정확도만 저장 |
| GPS 실패 출퇴근 허용 | QR 또는 버튼 클릭만으로 허용된 기록을 어떻게 리뷰할 것인가 | 허용하되 `GPS수신실패` 필터로 관리자 확인 |
| 초과근무수당 인정 | 어떤 기준으로 관리자가 수당 집계 대상으로 인정할 것인가 | 야근 신청 승인과 별도 `수당 인정` 체크 |
| 급여명세서 삭제 | 직원 요청 삭제를 허용할 것인가 | HR 관리자만 삭제 |

## 7. CTO 결론

대표 결정사항은 MVP에 반영 가능하다. 구현 복잡도도 과하지 않다.

고정 QR은 운영 편의성을 위해 유지한다. GPS 성공 시 정상 출퇴근, GPS 실패 시에는 `GPS수신실패`를 노티/마킹하고 QR 또는 버튼 클릭으로 출퇴근을 허용한다. 조기퇴근과 야근 상계는 급여 자동계산에 바로 연결하지 말고, 장부와 관리자 인정 기능으로 구현한다.

파일럿은 운영팀 4명으로 시작하고, 2주간 GPS 반경 300m, 고정 QR, GPS수신실패 허용, 선사용 휴가 충전, 조기퇴근-야근 상계, 관리자 출퇴근 보정 정책을 검증한 뒤 제작팀 5명으로 확대하는 것을 권장한다.
